# MongoDB

MongoDB support is live in this repo through both `@farming-labs/orm-mongo`
and `@farming-labs/orm-mongoose`.

Those packages do not try to pretend MongoDB is SQL. Instead, they reuse the
same schema contract and typed query API while translating calls into either
native `mongodb` collection operations or Mongoose-style reads, writes, and
session-aware transactions.

## What ships today

- a live `@farming-labs/orm-mongo` runtime package
- a live `@farming-labs/orm-mongoose` runtime package
- the shared schema DSL from `@farming-labs/orm`
- the same typed CRUD and relation query surface used by the SQL runtime

What does **not** ship today:

- a MongoDB generator target
- a Prisma-style Mongo runtime driver

## How it fits

The intended flow is:

- library authors define one schema in `@farming-labs/orm`
- library authors write one set of storage helpers against `createOrm(...)`
- MongoDB apps can bind `@farming-labs/orm-mongo`
- MongoDB apps bind `@farming-labs/orm-mongoose`

That means a helper like this:

```ts
export function createAuthStore(orm: typeof authOrm) {
  return {
    findUserByEmail(email: string) {
      return orm.user.findUnique({
        where: {
          email: email.trim().toLowerCase(),
        },
        select: {
          id: true,
          email: true,
          sessions: {
            select: {
              token: true,
              expiresAt: true,
            },
          },
        },
      });
    },
  };
}
```

can run unchanged whether the app chooses:

- memory for tests
- direct SQL for relational databases
- native MongoDB for apps using `mongodb`
- Mongoose for MongoDB-backed apps

## Native runtime example

```ts
import { createOrm } from "@farming-labs/orm";
import { createMongoDriver } from "@farming-labs/orm-mongo";
import { MongoClient } from "mongodb";
import { authSchema } from "./schema";

const client = new MongoClient(process.env.MONGODB_URL!);
await client.connect();

const orm = createOrm({
  schema: authSchema,
  driver: createMongoDriver({
    db: client.db("app"),
    client,
  }),
});
```

If the app already has Mongoose models and wants to keep them, use
`@farming-labs/orm-mongoose` instead:

```ts
import { createOrm } from "@farming-labs/orm";
import { createMongooseDriver } from "@farming-labs/orm-mongoose";
import mongoose from "mongoose";
import { authSchema } from "./schema";
import { AccountModel, ProfileModel, SessionModel, UserModel } from "./models";

const orm = createOrm({
  schema: authSchema,
  driver: createMongooseDriver<typeof authSchema>({
    models: {
      user: UserModel,
      profile: ProfileModel,
      session: SessionModel,
      account: AccountModel,
    },
    connection: mongoose.connection,
  }),
});
```

Once created, the runtime client is the same shape:

```ts
await orm.user.findUnique(...)
await orm.user.findMany(...)
await orm.user.create(...)
await orm.user.update(...)
await orm.user.upsert(...)
await orm.user.delete(...)
await orm.transaction(async (tx) => {
  // ...
})
```

## Field mapping and `_id`

MongoDB apps often want logical field names in library code but different
physical names in stored documents.

That is why the schema still matters:

```ts
const authSchema = defineSchema({
  user: model({
    table: "users",
    fields: {
      id: id().map("_id"),
      email: string(),
      emailVerified: boolean().map("email_verified"),
      createdAt: datetime().map("created_at"),
    },
  }),
});
```

With that mapping:

- library code still uses `id`, `emailVerified`, and `createdAt`
- the MongoDB and Mongoose runtimes read and write `_id`, `email_verified`, and `created_at`

## Relations in MongoDB-oriented apps

`@farming-labs/orm-mongo` and `@farming-labs/orm-mongoose` support the same
relation metadata as the other runtime drivers:

- `belongsTo`
- `hasOne`
- `hasMany`
- `manyToMany`

Relation loading is intentionally capability-aware. Instead of assuming SQL
joins, the driver loads related documents through follow-up queries and then
projects the nested `select` shape back into the typed result.

That means auth-style queries like this still work:

```ts
await orm.user.findUnique({
  where: { email: "ada@farminglabs.dev" },
  select: {
    id: true,
    profile: {
      select: {
        bio: true,
      },
    },
    sessions: {
      orderBy: {
        token: "desc",
      },
      take: 2,
      select: {
        token: true,
      },
    },
  },
});
```

## Transactions

If you provide `client` or `startSession`, `@farming-labs/orm-mongo` uses
MongoDB sessions for `orm.transaction(...)`.

If you provide `connection: mongoose.connection` or `startSession`,
`@farming-labs/orm-mongoose` uses Mongoose sessions for `orm.transaction(...)`.

If no session source is configured, the runtime still works, but transactional
workflows fall back to non-transactional execution.

That makes the package usable in:

- simple scripts
- lighter-weight apps
- test doubles and mocked model layers

while still supporting real session-aware transactions in production apps.

## Why Mongoose is its own capability profile

MongoDB is not just “another SQL dialect.”

Important differences include:

- document storage instead of relational tables
- no direct equivalent to SQL foreign keys
- different transaction/session constraints
- relation loading through explicit follow-up queries rather than assuming joins

So the right design is:

- one shared query API
- one shared schema contract
- separate runtime implementations per capability profile

That is exactly why the repo ships both `@farming-labs/orm-mongo` and
`@farming-labs/orm-mongoose` as MongoDB-specific runtime packages.
