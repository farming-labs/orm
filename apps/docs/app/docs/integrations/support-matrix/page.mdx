# Support Matrix

This page is the fastest way to answer:

- which stacks are supported today
- which helpers work with each stack
- which features are first-class, partial, or intentionally unsupported

If you want the detailed integration story for one stack, use the guides in
[Integrations](/docs/integrations). If you want the quick support answer, use
this page.

| `createDriverFromRuntime(...)` | Yes    | Yes     | Yes    | Yes     | Yes       | Yes | Yes     | Yes      | Yes       |

Important nuance:

- Firestore intentionally does not have SQL-style DDL push, so `pushSchema(...)`
  and `applySchema(...)` are no-op setup steps there rather than real schema
  application
- Supabase fits through the PostgreSQL path, not a separate runtime family

## Query and relation behavior

| Feature                                        | SQL / Drizzle / Kysely | Prisma  | TypeORM | Sequelize | MongoDB / Mongoose | Firestore |
| ---------------------------------------------- | ---------------------- | ------- | ------- | --------- | ------------------ | --------- |
| `create`, `update`, `delete`, `find*`, `count` | Yes                    | Yes     | Yes     | Yes       | Yes                | Yes       |
| transactions                                   | Yes                    | Yes     | Yes     | Yes       | Yes                | Yes       |
| compound unique lookups                        | Yes                    | Yes     | Yes     | Yes       | Yes                | Yes       |
| native relation loading                        | Partial                | Partial | Partial | Partial   | No                 | No        |
| fallback relation resolver                     | Yes                    | Yes     | Yes     | Yes       | Yes                | Yes       |

“Partial” here means the native path covers the safest direct branches first,
and more complex relation shapes still fall back to the shared resolver.

That is why stacks without joins or SQL-style transactions can still work
correctly through the same higher-level storage layer: the ORM handles the
necessary consecutive reads and write orchestration instead of requiring the
consumer package to fork its logic.

## Scalar and schema support

| Feature                                            | SQL-family / Prisma / TypeORM / Sequelize | MongoDB / Mongoose | Firestore |
| -------------------------------------------------- | ----------------------------------------- | ------------------ | --------- |
| `integer()`                                        | Yes                                       | Yes                | Yes       |
| `json()`                                           | Yes                                       | Yes                | Yes       |
| `enumeration()`                                    | Yes                                       | Yes                | Yes       |
| `bigint()`                                         | Yes                                       | Yes                | Yes       |
| `decimal()`                                        | Yes                                       | Yes                | Yes       |
| manual numeric IDs                                 | Yes                                       | Yes                | Yes       |
| generated numeric IDs                              | Yes                                       | No                 | No        |
| schema namespaces via `tableName(..., { schema })` | PostgreSQL-family only                    | No                 | No        |

Important boundaries:

- generated integer IDs are first-class on SQL-family runtimes, Prisma,
  TypeORM, Sequelize, and memory
- MongoDB, Mongoose, and Firestore keep numeric IDs manual-only
- schema-qualified tables are Postgres-style only
- flat strings like `"auth.users"` are intentionally rejected; use
  `tableName("users", { schema: "auth" })`

## Capabilities and errors

All live runtimes expose:

- `orm.$driver.capabilities`
- normalized `OrmError` values

That means higher-level libraries can inspect:

- transactions
- numeric ID behavior
- namespace support
- upsert behavior
- returning behavior
- text comparison behavior
- relation-loading capability

without branching on backend-specific error codes or adapter-specific metadata.
