# Integrations

Farming Labs ORM has two layers:

- `@farming-labs/orm` defines the schema and query API
- runtime packages execute that API against the app’s actual stack

## Support matrix

| Layer               | PostgreSQL | MySQL | SQLite | MongoDB | Notes                                     |
| ------------------- | ---------- | ----- | ------ | ------- | ----------------------------------------- |
| Prisma generation   | Yes        | Yes   | Yes    | No      | `postgresql`, `mysql`, `sqlite` providers |
| Drizzle generation  | Yes        | Yes   | Yes    | No      | `pg`, `mysql`, `sqlite` dialects          |
| Safe SQL generation | Yes        | Yes   | Yes    | No      | `postgres`, `mysql`, `sqlite` dialects    |
| Memory runtime      | N/A        | N/A   | N/A    | N/A     | In-memory driver for tests and demos      |
| Prisma runtime      | Yes        | Yes   | Yes    | No      | `@farming-labs/orm-prisma`                |
| Drizzle runtime     | Yes        | Yes   | Yes    | No      | `@farming-labs/orm-drizzle`               |
| Direct SQL runtime  | Yes        | Yes   | Yes    | No      | `@farming-labs/orm-sql`                   |
| MongoDB runtime     | No         | No    | No     | Yes     | `@farming-labs/orm-mongo`                 |
| Mongoose runtime    | No         | No    | No     | Yes     | `@farming-labs/orm-mongoose`              |
| Kysely runtime      | Yes        | Yes   | Yes    | No      | `@farming-labs/orm-kysely`                |

## `integer()` and `json()` coverage

matrix:

| Runtime / stack    | PostgreSQL | MySQL | SQLite | MongoDB | `integer()` | `json()` |
| ------------------ | ---------- | ----- | ------ | ------- | ----------- | -------- |
| Memory runtime     | N/A        | N/A   | N/A    | N/A     | Yes         | Yes      |
| Prisma runtime     | Yes        | Yes   | Yes    | No      | Yes         | Yes      |
| Drizzle runtime    | Yes        | Yes   | Yes    | No      | Yes         | Yes      |
| Kysely runtime     | Yes        | Yes   | Yes    | No      | Yes         | Yes      |
| Direct SQL runtime | Yes        | Yes   | Yes    | No      | Yes         | Yes      |
| MongoDB runtime    | No         | No    | No     | Yes     | Yes         | Yes      |
| Mongoose runtime   | No         | No    | No     | Yes     | Yes         | Yes      |

Those checks are not generator-only. They run the live runtime query layer
against real local database instances and real client stacks for each supported
dialect combination.

Use this command to run the whole local matrix:

```bash
pnpm test:local
```

## Native relation loading

The current runtime uses 2 strategies:

- native relation loading where the backend can translate the query shape directly
- fallback relation resolution for shapes that still need the shared runtime layer

Today that means:

| Runtime / stack    | Native path today                                                                                                               | Fallback today                                                           |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| Prisma runtime     | direct nested Prisma `select` for supported relation branches, including simple explicit join-table `manyToMany(...)` traversal | relation branches with extra relation-level filters, ordering, or paging |
| Drizzle runtime    | inherited SQL single-query loading for singular chains and simple collection branches without relation-level modifiers          | relation branches with extra relation-level filters, ordering, or paging |
| Kysely runtime     | inherited SQL single-query loading for singular chains and simple collection branches without relation-level modifiers          | relation branches with extra relation-level filters, ordering, or paging |
| Direct SQL runtime | single SQL statement for singular chains and simple collection branches without relation-level modifiers                        | relation branches with extra relation-level filters, ordering, or paging |
| Mongo / Mongoose   | not yet                                                                                                                         | shared relation resolver                                                 |

The important nuance is that the fallback column does **not** mean the backend
is fundamentally missing relation support.

- SQL-backed runtimes can support much richer native joins than the first
  singular path implemented here
- Prisma can support more native nested relation shapes than the current direct
  branch translation
- MongoDB and Mongoose can grow a native aggregation / `$lookup` path later

So the current matrix is best read as: **what this repo natively translates
today**, not **the theoretical limit of the backend**.

## How to choose

- Use **Prisma generation** if your app already owns `schema.prisma`
- Use **Drizzle generation** if your app wants generated Drizzle tables
- Use **safe SQL generation** if you want SQL files directly
- Use **Prisma runtime** if your app already uses `PrismaClient`
- Use **Drizzle runtime** if your app already uses Drizzle
- Use **Kysely runtime** if your app already uses Kysely
- Use **direct SQL runtime** if you want to talk to SQLite, MySQL, or PostgreSQL directly
- Use **MongoDB runtime** if your app uses the native `mongodb` client
- Use **Mongoose runtime** if your app already uses Mongoose models

## What “supported” means

When a runtime is marked live, the unified ORM methods already translate into
real calls for that stack.

Examples:

- `@farming-labs/orm-prisma` translates into Prisma delegate calls
- `@farming-labs/orm-drizzle` translates into Drizzle-backed queries
- `@farming-labs/orm-kysely` translates into Kysely-backed queries
- `@farming-labs/orm-sql` translates into direct SQL operations
- `@farming-labs/orm-mongo` translates into native `mongodb` collection operations
- `@farming-labs/orm-mongoose` translates into Mongoose model operations
