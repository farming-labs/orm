# Integrations

This section explains how Farming Labs ORM fits into real application stacks.

The most important idea is this:

- `@farming-labs/orm` is the schema contract
- `@farming-labs/orm-cli` generates artifacts for app-side tools
- `@farming-labs/orm-sql` runs the unified runtime directly against SQL backends
- `@farming-labs/orm-mongoose` runs the same runtime contract against Mongoose models

So when people ask “does this support Prisma or Drizzle?” the honest answer is:

- **yes for generation today**
- **yes for live runtime execution today through memory, direct SQL, and Mongoose drivers**
- **not yet for live Prisma, Drizzle, or Kysely runtime execution in this repo**

## Current support matrix

| Layer                  | PostgreSQL | MySQL   | SQLite  | MongoDB | Notes                                                           |
| ---------------------- | ---------- | ------- | ------- | ------- | --------------------------------------------------------------- |
| Prisma generation      | Yes        | Yes     | Yes     | No      | Uses Prisma providers `postgresql`, `mysql`, `sqlite`           |
| Drizzle generation     | Yes        | Yes     | Yes     | No      | Uses Drizzle dialects `pg`, `mysql`, `sqlite`                   |
| Safe SQL generation    | Yes        | Yes     | Yes     | No      | Uses SQL dialects `postgres`, `mysql`, `sqlite`                 |
| Memory runtime         | N/A        | N/A     | N/A     | N/A     | In-memory driver for tests, demos, and docs                     |
| Direct SQL runtime     | Yes        | Yes     | Yes     | No      | `@farming-labs/orm-sql` for SQLite, MySQL, and PostgreSQL       |
| Mongoose runtime       | No         | No      | No      | Yes     | `@farming-labs/orm-mongoose` for MongoDB-oriented apps          |
| Prisma runtime driver  | Planned    | Planned | Planned | Later   | Future package; Mongo via Prisma is not part of this repo today |
| Drizzle runtime driver | Planned    | Planned | Planned | No      | Future package                                                  |
| Kysely runtime driver  | Planned    | Planned | Planned | No      | Future package                                                  |

## How to think about “fit”

### Library author side

The library author writes:

- one schema
- one docs story
- one set of runtime helper ideas

### App side

The consuming app chooses:

- Prisma generation
- Drizzle generation
- safe SQL generation
- or direct SQL runtime with `@farming-labs/orm-sql`
- or Mongoose runtime with `@farming-labs/orm-mongoose`

That means the application keeps control over its own persistence toolchain.

## How method translation works

Every model method follows the same path:

```ts
orm.user.findUnique(...)
orm.user.findMany(...)
orm.user.create(...)
orm.user.update(...)
orm.user.upsert(...)
```

Those calls are handed to the active runtime driver.

- In `@farming-labs/orm-sql`, they already compile into SQLite, MySQL, and PostgreSQL queries.
- In `@farming-labs/orm-mongoose`, they already compile into Mongoose filters, writes, and transactions.
- In the memory driver, they execute against in-memory rows for tests and demos.
- In future Prisma, Drizzle, and Kysely drivers, those same methods would translate into the target client’s operations instead of SQL strings.

For generated Prisma and Drizzle output, direct relation definitions are also covered
today:

- `belongsTo`
- `hasOne`
- `hasMany`

Explicit join-table `manyToMany` relationships still flow through the join model in
generated Prisma and Drizzle output.

## What “supported” means in practice

When we say Prisma or Drizzle is supported **today**, we mean:

- the schema can be rendered into those target formats
- the generated files can then be consumed by those ecosystems

It does **not** mean that this repo already ships a live runtime executor backed
by `PrismaClient` or `drizzle-orm` at runtime.

It **does** mean the repo now ships live runtime packages for:

- SQLite
- MySQL
- PostgreSQL via `pg.Pool`
- MongoDB via Mongoose models and sessions

That distinction matters a lot in the docs, because otherwise “support” becomes
too vague to be useful.
