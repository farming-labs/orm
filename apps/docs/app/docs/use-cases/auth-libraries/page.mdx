# Auth Libraries

Auth packages are one of the clearest fits for Farming Labs ORM.

## Why auth packages suffer

They usually need to define the same storage ideas repeatedly:

- users
- sessions
- accounts
- verification records
- memberships
- plugin-owned models

And then they often repeat that work again for:

- Prisma
- Drizzle
- raw SQL examples
- docs and setup guides

## One schema instead

```ts
export const authSchema = defineSchema({
  user: model({
    table: "users",
    fields: {
      id: id(),
      email: string().unique(),
      name: string(),
      createdAt: datetime().defaultNow(),
    },
    relations: {
      sessions: hasMany("session", { foreignKey: "userId" }),
      accounts: hasMany("account", { foreignKey: "userId" }),
    },
  }),
  session: model({
    table: "sessions",
    fields: {
      id: id(),
      userId: string().references("user.id"),
      token: string().unique(),
      expiresAt: datetime(),
    },
    relations: {
      user: belongsTo("user", { foreignKey: "userId" }),
    },
  }),
  account: model({
    table: "accounts",
    fields: {
      id: id(),
      userId: string().references("user.id"),
      provider: string(),
      accountId: string(),
    },
    relations: {
      user: belongsTo("user", { foreignKey: "userId" }),
    },
  }),
});
```

## App-side generation

```ts
export default defineConfig({
  schemas: [authSchema],
  targets: {
    prisma: {
      out: "./generated/prisma/schema.prisma",
      provider: "postgresql",
    },
  },
});
```

## Runtime helper example

```ts
const normalizeEmail = (email: string) => email.trim().toLowerCase();

export function createAuthStore(orm: typeof authOrm) {
  return {
    findUserByEmail(email: string) {
      return orm.user.findUnique({
        where: {
          email: normalizeEmail(email),
        },
        select: {
          id: true,
          email: true,
          sessions: {
            select: {
              token: true,
              expiresAt: true,
            },
          },
          accounts: {
            select: {
              provider: true,
              accountId: true,
            },
          },
        },
      });
    },

    createOAuthUser(input: {
      name: string;
      email: string;
      provider: string;
      accountId: string;
    }) {
      return orm.transaction(async (tx) => {
        const user = await tx.user.create({
          data: {
            name: input.name,
            email: normalizeEmail(input.email),
          },
          select: {
            id: true,
            email: true,
          },
        });

        const account = await tx.account.create({
          data: {
            userId: user.id,
            provider: input.provider,
            accountId: input.accountId,
          },
          select: {
            provider: true,
            accountId: true,
          },
        });

        return { user, account };
      });
    },

    rotateSession(input: { userId: string; token: string; expiresAt: Date }) {
      return orm.session.upsert({
        where: {
          token: input.token,
        },
        create: {
          userId: input.userId,
          token: input.token,
          expiresAt: input.expiresAt,
        },
        update: {
          expiresAt: input.expiresAt,
        },
        select: {
          token: true,
          expiresAt: true,
        },
      });
    },

    invalidateUserSessions(userId: string) {
      return orm.session.deleteMany({
        where: {
          userId,
        },
      });
    },
  };
}
```

If a library normalizes emails, it should use the same helper on both writes
and lookups. Lowercasing only at query time can miss rows that were stored in a
different case.

That gives an auth library one place to implement:

- load a user by email, id, or token
- rotate sessions
- invalidate sessions
- link or sync provider accounts
- count related records for dashboards or plugin rules

The important part is that the auth package writes those helpers once against
the unified runtime contract. The consumer app chooses the eventual target
stack.

## What this removes

- storage docs duplicated per ORM
- package-owned schema drift across stacks
- hand-maintained adapter surfaces for every consumer stack

It does not magically remove all complexity, but it centralizes the part that is
actually shared.
