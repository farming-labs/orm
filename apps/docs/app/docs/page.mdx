# Overview

Farming Labs ORM is built for packages and platforms that want one schema
contract, one docs story, and one generation path instead of maintaining a
different storage adapter for every ORM.

## What this project is

- A schema DSL in `@farming-labs/orm`
- A typed client surface built around `createOrm(...)`
- A memory runtime driver that already supports nested relation selection
- A Prisma runtime package for apps that already use `PrismaClient`
- A direct SQL runtime package for SQLite, MySQL, and PostgreSQL
- A Mongoose runtime package for MongoDB-backed apps
- A generator CLI that writes Prisma, Drizzle, and safe SQL output

## What this project is not

- It is not a second full ORM with its own migrations, query planner, and SQL execution engine.
- It does not yet ship live Drizzle or Kysely runtime drivers in this repo.
- Generated Drizzle output still does not imply a live Drizzle runtime executor in this repo.
- Join-table-backed `manyToMany(...)` relationships still stay explicit in generated Prisma and Drizzle output.

## End-to-end mental model

```ts
import { defineSchema, model, id, string } from "@farming-labs/orm";
import { defineConfig } from "@farming-labs/orm-cli";

export const authSchema = defineSchema({
  user: model({
    table: "users",
    fields: {
      id: id(),
      email: string().unique(),
      name: string(),
    },
  }),
});

export default defineConfig({
  schemas: [authSchema],
  targets: {
    prisma: { out: "./generated/prisma/schema.prisma", provider: "postgresql" },
    drizzle: { out: "./generated/drizzle/schema.ts", dialect: "pg" },
    sql: { out: "./generated/sql/0001_init.sql", dialect: "postgres" },
  },
});
```

That same schema can then also drive the typed client surface:

```ts
import { createMemoryDriver, createOrm } from "@farming-labs/orm";

const orm = createOrm({
  schema: authSchema,
  driver: createMemoryDriver(),
});

await orm.user.create({
  data: {
    email: "ada@farminglabs.dev",
    name: "Ada",
  },
});
```

## What ships today

| Layer                        | Status | Notes                                                            |
| ---------------------------- | ------ | ---------------------------------------------------------------- |
| `@farming-labs/orm`          | Live   | Schema DSL, manifest, typed client, generators, memory runtime   |
| `@farming-labs/orm-cli`      | Live   | Loads schemas directly and writes target output                  |
| `@farming-labs/orm-prisma`   | Live   | Prisma runtime driver that translates the unified API into Prisma delegates |
| `@farming-labs/orm-sql`      | Live   | Direct SQL runtime for SQLite, MySQL, and PostgreSQL pools       |
| `@farming-labs/orm-mongoose` | Live   | MongoDB / Mongoose runtime using the same typed query API        |
| Memory runtime               | Live   | Supports typed CRUD, nested relation selection, and transactions |
| Prisma runtime               | Live   | Uses the same query API against `PrismaClient` delegates         |
| Direct SQL runtime           | Live   | Uses the same query API against SQLite, MySQL, and `pg.Pool`     |
| Mongoose runtime             | Live   | Uses the same query API against Mongoose models and sessions     |
| Prisma generation            | Live   | PostgreSQL, MySQL, and SQLite providers with direct relation output |
| Drizzle generation           | Live   | `pg`, `mysql`, and `sqlite` dialects with direct relation helpers  |
| Safe SQL generation          | Live   | PostgreSQL, MySQL, and SQLite                                    |
| Runtime driver packages      | Mixed  | Prisma is live; Drizzle and Kysely are the next runtime layers   |

## Current architecture

- The schema DSL defines models, fields, references, descriptions, and higher-level relation metadata.
- The manifest layer turns that schema into normalized table and column information.
- The generators consume manifest-level field data such as mapped column names, uniqueness, defaults, and references.
- The Prisma and Drizzle generators now also emit direct `belongsTo`, `hasOne`, and `hasMany` relation shapes from that same schema metadata.
- The Prisma runtime, memory driver, direct SQL runtime, and Mongoose runtime consume both field data and relation metadata so nested `select` queries work today, including explicit join-table `manyToMany(...)`.
- The CLI merges multiple schema objects and refuses duplicate model names.

## Where this fits best

- Auth-like packages that want one storage contract and many consuming stacks
- Billing and org kits that need reusable models and generated output
- Internal platform modules shared across many apps or services
- Greenfield teams that want a schema-first start without locking into one app-side ORM shape immediately
